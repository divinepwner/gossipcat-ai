# Session Memory — Orchestrator Continuity Across Sessions

**Date:** 2026-03-29
**Status:** Spec v2 (updated after 3-agent consensus review)

---

## The Problem

Every gossipcat session starts from zero. The orchestrator (the LLM that dispatches tasks, chooses agents, interprets results) has no memory of prior sessions. It doesn't know:

- What the team accomplished yesterday
- Which approaches failed and why
- Which agents are strong/weak at specific tasks
- What's in progress (specs, half-implemented features)
- User preferences expressed in prior sessions

This makes the orchestrator the dumbest part of a system that's getting smarter everywhere else. Agents have cognitive memory. Performance signals accumulate. Skills evolve. But the thing directing all of it starts fresh every time.

## The Thesis

**Gossipcat is the only multi-agent system where the orchestrator gets smarter across sessions.** CrewAI, AutoGen, LangGraph — none have session-level orchestrator memory. This is gossipcat's differentiator: **the mesh remembers.**

## Design

### Core Principle: The orchestrator is a virtual agent

Agent ID: `_project` (reserved — setup/skill-index validation must reject this ID for real agents).

Uses the same memory infrastructure as real agents:

```
.gossip/agents/_project/memory/
├── MEMORY.md              ← Project memory index (loaded into bootstrap)
├── tasks.jsonl            ← Session entries (warmth-scored, compacted)
├── archive.jsonl          ← Cold sessions evicted by compactor
├── session-gossip.jsonl   ← Persisted session gossip (survives crashes)
└── knowledge/
    ├── 2026-03-29T22-session.md   ← Cognitive session summary
    ├── 2026-03-28T19-consensus.md ← Key consensus insights
    └── ...                         ← Max 10, warmth-sorted pruning
```

**Why reuse agent infrastructure?** MemoryWriter, MemoryCompactor, AgentMemoryReader all work. Warmth-based decay works. Cognitive summaries work. Bootstrap generator already knows how to read agent memory.

**Modifications to existing infrastructure required:**
1. Knowledge file pruner must use warmth-sorted eviction (not pure recency) for `_project`
2. Bootstrap reader must use warmth-only selection (not relevance matching) for `_project`
3. Session summary needs its own write path (not reuse `generateCognitiveSummary` with its 1500-char limit)

### What gets written and when

#### 1. After each consensus run (automatic)

When `runConsensus()` completes with findings, write a knowledge entry for `_project`:

```
What was reviewed, by which agents
How many confirmed/disputed/unverified
Key confirmed findings (top 3 by confidence)
Agent observations (who agreed with whom, who was disputed)
```

Generated by the utility LLM with a consensus-specific prompt.

**Trigger:** Inside `DispatchPipeline.runConsensus()`, after signal writing (line ~978). Fire-and-forget with `.catch()` — accept data loss risk over latency.
**Cost:** One utility LLM call per consensus run (~500 tokens).
**Data:** All needed data (`results`, `consensusReport`) is in scope at the insertion point.

Additionally: `DispatchPipeline` must cache consensus reports in a new `sessionConsensusHistory: ConsensusReport[]` array, populated at the end of `runConsensus()`. This feeds into `gossip_session_save`.

#### 2. Session save (user-triggered MCP tool)

New tool: `gossip_session_save`

```typescript
gossip_session_save(notes?: string)
```

Generates a comprehensive session summary from:
- Session gossip entries (from persisted `session-gossip.jsonl`, not just in-memory)
- Consensus runs this session (from `sessionConsensusHistory` array)
- Agent performance changes (scores at session start vs now)
- Skill index changes (new bindings, toggles)
- Git log since session start (capped at `--max-count=50`)
- User notes (optional freeform context)

The LLM synthesizes all of this into a cognitive summary.

**Importance:** Session save entries always write with `importance: 0.95` directly, bypassing `deriveImportance()` (which would undershoot with synthetic scores).

**LLM failure fallback:** If the LLM call fails, save the raw data sources as structured markdown (git log, gossip entries, consensus counts). Not as rich, but ensures data is not lost.

**Trigger:** User calls `gossip_session_save()` before ending session.
**Discoverability:** After 5+ task completions or any consensus run, append to collect output: `"💡 Active session — consider gossip_session_save() to persist what you've learned."`
**Also:** Prints reminder: "Session saved. Next session, gossip_bootstrap() will load this context automatically."

#### 3. Session gossip persistence (automatic, crash-safe)

Current `sessionGossip` is in-memory only — lost on crash. Fix: persist to disk.

After each `summarizeAndStoreGossip()` call, also append to `.gossip/agents/_project/memory/session-gossip.jsonl`. On boot, the file is cleared (new session). On `gossip_session_save`, it's read as a data source.

This ensures `gossip_session_save` has data even after an MCP server restart mid-session.

### How it loads into the next session

`BootstrapGenerator.generate()` reads `_project` memory with a **warmth-only reader** (no relevance filtering):

```typescript
private readProjectMemory(): string | null {
  const memDir = join(this.projectRoot, '.gossip', 'agents', '_project', 'memory');
  if (!existsSync(memDir)) return null;

  const knowledgeDir = join(memDir, 'knowledge');
  if (!existsSync(knowledgeDir)) return null;

  // Read all knowledge files, sort by warmth (importance × recency), take top 3
  const files = readdirSync(knowledgeDir).filter(f => f.endsWith('.md'));
  const scored = files.map(f => {
    const content = readFileSync(join(knowledgeDir, f), 'utf-8');
    const importance = parseFloat(content.match(/importance:\s*([\d.]+)/)?.[1] ?? '0.5');
    const timestamp = f.slice(0, 19).replace(/-/g, (m, i) => i > 9 ? ':' : '-'); // approx
    const warmth = calculateWarmth(importance, timestamp);
    return { file: f, content, warmth };
  });

  scored.sort((a, b) => b.warmth - a.warmth);

  // Return body content of top entries, capped at 2500 chars
  const bodies = scored.slice(0, 3).map(s => {
    const bodyStart = s.content.indexOf('---', 4);
    return bodyStart !== -1 ? s.content.slice(bodyStart + 3).trim() : '';
  }).filter(Boolean);

  return bodies.join('\n\n---\n\n').slice(0, 2500) || null;
}
```

**Why warmth-only?** `AgentMemoryReader.loadMemory('_project', '')` with an empty task string produces zero relevance for all files (verified in agent-memory.ts:55-60 — empty taskWords means no word matches). A warmth-only reader selects the most recent + important entries without needing a task context.

**Cap:** 2500 chars (bumped from 2000 based on agent feedback that complex sessions need more space).

The bootstrap prompt then includes:

```markdown
## Session Context

[warmth-sorted knowledge file content here]
```

### Evergreen knowledge (pinned entries)

**Problem identified by agents:** Warmth-based decay will eventually evict critical lessons. A finding like "STOP_WORDS don't work" from 60 days ago has warmth 0.45 — it will be evicted when 10 newer sessions are written.

**Fix:** Support a `pinned: true` frontmatter field in knowledge files:

```markdown
---
name: STOP_WORDS approach failed
importance: 0.95
pinned: true
---
```

Pinned knowledge files:
- Are never evicted by the knowledge pruner
- Always get warmth = 1.0 in the reader (bypass decay)
- Count against the MAX_KNOWLEDGE_FILES limit (so max 10 total, pinned + unpinned)

**Who pins?** The `gossip_session_save` prompt instructs the LLM to mark the summary as `pinned: true` if it contains "never do this again" lessons. Additionally, users can manually edit files to add `pinned: true`.

**Implementation:** 3 lines in the knowledge pruner (skip files with `pinned: true`) + 3 lines in the reader (override warmth to 1.0 for pinned).

### Knowledge file pruning: warmth-sorted for `_project`

**Problem identified by agents:** The current knowledge pruner (`memory-writer.ts:101-110`) sorts files alphabetically (by timestamp prefix) and removes the oldest. This ignores importance entirely — a high-value lesson from 60 days ago gets deleted before a low-value entry from yesterday.

**Fix for `_project`:** When pruning knowledge files, sort by warmth (same calculation as compactor) and remove the lowest-warmth entries. Pinned files are exempt.

This only applies to `_project` knowledge files. Regular agent knowledge files keep the existing recency-based pruning (their knowledge is more ephemeral — task-specific context, not cross-session lessons).

### Session summary write path

**Problem identified by agents:** `generateCognitiveSummary` in `memory-writer.ts` truncates output to 1500 chars (~250-300 words). This is too short for a comprehensive session summary.

**Fix:** `gossip_session_save` uses a dedicated write method `writeSessionSummary()` on MemoryWriter, with:
- Output cap: 3000 chars (~500-600 words)
- Session-specific prompt (5 sections, warning-oriented)
- Direct importance assignment (0.95, not derived from scores)
- Optional `pinned: true` based on LLM judgment

This method calls the same `summaryLlm.generate()` but with different prompt and output limits.

### The prompt for session summary generation

```
You are writing a project memory entry that will be loaded into the
orchestrator's context at the start of the next session. This context
helps the orchestrator make better decisions about agent dispatch,
task planning, and avoiding past mistakes.

Write as a briefing for a new team lead taking over. Focus on:

1. WHAT SHIPPED — concrete deliverables, not process descriptions.
   Name the features, cite file paths if relevant.

2. WHAT FAILED AND WHY — approaches that didn't work. Be specific
   about why, so future sessions don't retry them.
   Format: "We tried X because Y. It failed because Z. The fix was W."

3. AGENT OBSERVATIONS — which agents are reliable for what, who
   hallucinates, who finds things others miss. Back with data.

4. IN PROGRESS — specs in review, half-built features, known bugs.
   Include file paths and what specifically needs to happen next.

5. USER PREFERENCES — how the user likes to work (e.g., "always
   run multi-agent review before merging", "prefers conservative
   thresholds over aggressive optimization").

Rules:
- Max 500 words
- No preamble. Start with "## What shipped"
- Cite file paths when referencing code or specs
- Include specific numbers (commit count, finding count, test count)
- Warnings > accomplishments — it's more useful to know what NOT to do
- If ANY section contains a critical "never do this again" lesson, add
  "pinned: true" to the frontmatter
```

### MCP tool interface

```typescript
gossip_session_save(notes?: string)
// notes: optional freeform user context ("we're focusing on security hardening")
// Returns: the generated summary text
// Side effects:
//   1. Reads session-gossip.jsonl + sessionConsensusHistory + git log + scores
//   2. Calls utility LLM to generate cognitive summary (fallback: raw data save)
//   3. Writes knowledge entry to .gossip/agents/_project/memory/knowledge/
//   4. Writes task entry to .gossip/agents/_project/memory/tasks.jsonl (importance: 0.95)
//   5. Rebuilds .gossip/agents/_project/memory/MEMORY.md index
//   6. Clears session-gossip.jsonl (consumed)
//   7. Prints reminder about gossip_bootstrap() for next session
```

### Reserved ID: `_project`

The `_project` agent ID must be reserved and rejected in:
- `gossip_setup` agent creation (`mcp-server-sdk.ts`)
- `SkillIndex.validateAgentId()` (`skill-index.ts`)
- `config.json` validation on load

Add `'_project'` to `DANGEROUS_KEYS` in `skill-index.ts` and add equivalent validation in setup.

### Data sources for summary generation

| Source | What it provides | How to access | Crash-safe? |
|--------|-----------------|---------------|-------------|
| Session gossip | Task summaries from this session | `.gossip/agents/_project/memory/session-gossip.jsonl` | Yes (persisted to disk) |
| Consensus history | Findings, agent attributions | `pipeline.sessionConsensusHistory[]` (in-memory) | Partial (auto-writes survive, array doesn't) |
| Agent performance | Signal counts, score changes | `PerformanceReader.getScores()` | Yes (JSONL on disk) |
| Skill index | Binding changes this session | Compare index mtime vs session start | Yes (JSON on disk) |
| Git log | Commits since session start | `git log --oneline --max-count=50 --since="<boot>"` | Yes (git) |
| User notes | Freeform context | Passed to the tool | N/A |

## Non-goals (v1)

- **Auto-detect session end** — MCP has no shutdown hook. User triggers `gossip_session_save` manually. Discoverability reminder mitigates forgetting.
- **Cross-project memory** — each project has its own `_project` agent. No sharing.
- **Memory editing UI** — the dashboard (future) could show session history, but not edit it.
- **Team conflict resolution** — if two users save simultaneously, both files persist (timestamped filenames). Bootstrap loads the most recent by warmth.

## Risk

**Low.** Reuses existing memory infrastructure with targeted modifications. The only new write paths are the session summary, session gossip persistence, and consensus auto-write. Failure modes:
- LLM produces bad summary → fallback saves raw data; bad file fades via warmth decay
- LLM fails entirely → raw data fallback, no data loss
- Server crash → session gossip persisted to disk, consensus auto-writes survive, only `sessionConsensusHistory` array is lost

**Privacy note:** Session summaries contain information about what was worked on (file paths, feature names, agent observations). This stays in `.gossip/` which is gitignored (verified: `.gitignore:7`). No data leaves the machine unless the user explicitly shares it.
